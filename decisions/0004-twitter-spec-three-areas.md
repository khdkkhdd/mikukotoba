# 트위터 명세를 트윗/유저/트렌드 3영역으로 구성

Status: accepted
Date: 2026-02-19

## Context

트위터(X) 번역 기능 명세를 작성하면서, 번역 대상이 트윗 본문·유저 정보·트렌딩 토픽으로 명확히 나뉘고, 각 영역의 표시 방식이 서로 다름을 확인했다. 유튜브처럼 자막/페이지로 2분할하는 방식이 적합하지 않아, 트위터에 맞는 구조가 필요했다.

## Decision

`docs/twitter_feature.md` 하나의 문서 안에서 **트윗**(본문, 카드, 투표), **유저 정보**(이름, 바이오, 위치, 팔로워 셀), **트렌딩 토픽**을 3영역으로 나누어 기술한다. 또한 요소의 성격에 따라 표시 방식을 차별화한다: 트윗 본문은 모드를 따르고, 짧은 텍스트(유저명·위치)는 항상 호버, 투표·트렌드는 괄호 형식을 사용한다.

## Consequences

### Positive
- 트위터 특유의 콘텐츠 구조(트윗 중심 + 소셜 메타 + 탐색)를 자연스럽게 반영
- 요소별 표시 방식 차이가 명확히 드러나, 리팩토링 시 각 핸들러의 역할이 분명함
- 유튜브 명세와 동일한 "사이트별 한 문서" 원칙 유지

### Negative
- 요소 유형이 많아(트윗 본문, 카드, 투표, 유저명, 바이오, 위치, 팔로워 셀, 트렌드) 문서가 다소 장황함
- 같은 "호버 팝업" 방식이 여러 요소에 반복 언급됨

## Alternatives Considered

- **페이지 단위 구성** (홈/프로필/검색 등): 같은 트윗이 여러 페이지에 동일하게 렌더링되므로 페이지별로 나누면 중복이 과도. Rejected.
- **표시 방식 단위 구성** (호버/인라인/괄호): 사용자가 "트윗은 어떻게 번역되나?"를 찾기 어려워짐. 콘텐츠 중심이 아닌 UI 중심 구성은 직관적이지 않음. Rejected.
